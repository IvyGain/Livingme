# Lark Base セキュリティ調査レポート

**調査日**: 2026-03-21
**対象**: Lark Suite / Lark Base (Bitable) / Lark Drive
**目的**: アートライフ会員サイトの CMS として Lark Base を採用することの安全性確認

---

## 結論（先に読んでください）

| 判定 | 理由 |
|------|------|
| **コンテンツデータ（イベント・アーカイブ情報など）の保存：⚠️ 条件付き可** | 技術的な暗号化は実装済み。ただし親会社 ByteDance（中国）によるデータアクセスリスクあり |
| **個人情報・会員情報の保存：❌ 非推奨** | 氏名・メール・決済情報など個人を特定できる情報を Lark に置くのは避けるべき |

> **現在の設計（PostgreSQL に個人情報、Lark に公開コンテンツ）は正しい方向性です。**
> ただし、Lark 側に保存するデータの範囲について十分な注意が必要です。

---

## 1. データ暗号化

**✅ 技術的な実装は十分**

| 種別 | 方式 | 詳細 |
|------|------|------|
| 保存時暗号化（At Rest） | AES-256 | KMS（鍵管理）+ HSM（ハードウェアセキュリティモジュール）で管理 |
| 転送時暗号化（In Transit） | HTTPS / TLS | 2048-bit RSA 証明書使用 |
| 鍵管理 | マルチパーティ制御 | 複数の管理者による分散鍵管理。単一の管理者が単独でアクセスできない |

技術的な暗号化レベルは AWS S3 / Google Cloud Storage などと同等水準です。

## 2. データ保存場所（データセンター）

**✅ 日本リージョンあり**

| リージョン | 場所 |
|-----------|------|
| 🇺🇸 US | アメリカ（AWS） |
| 🇯🇵 Japan | 日本（AWS） |
| 🇸🇬 Singapore | シンガポール（AWS） |

Lark は 2023年11月に AWS との全面移行を発表。全インフラを AWS 上に構築。

**⚠️ 注意点**：日本リージョンが「存在する」のは事実ですが、コミュニティ・個人の Workspace がどのリージョンに割り当てられるかは Workspace オーナーが確認・指定する必要があります。

## 4. 最大の懸念点：ByteDance（親会社）リスク

**⚠️ 高リスク ― 技術的セキュリティとは別次元の問題**

Lark の親会社は **ByteDance（字節跳動）** ── TikTok を運営する中国企業です。

### 2024年に判明した事実

2024年に米国司法省（DOJ）による調査で以下が明らかになりました：

> **TikTok の米国ユーザーデータが、Lark（ByteDance 社内コミュニケーションツール）を経由して中国の ByteDance スタッフにアクセスされていた。**

Lark は「国際版（Lark）と中国版（Feishu）のデータは完全に分離されている」と主張していますが、米司法省はこの分離が不十分と判断しています。

### 中国の国家情報法

中国の「国家情報法」（2017年制定）は、中国企業に対して政府機関からの要請があった場合にデータを提供する義務を課しています。ByteDance が中国企業である以上、この法律の適用対象となり得ます。

### アートライフへの影響度評価

| 保存するデータ | リスク |
|--------------|--------|
| イベント情報・アーカイブのタイトル・説明 | 低（公開情報のため） |
| 会員のメールアドレス・氏名 | **高（個人情報のため保存しないこと）** |
| 動画コンテンツ | 中（コンテンツ漏洩リスク） |
| ジャーナル（会員の個人日記） | **高（PostgreSQL で管理しており Lark に置かないこと）** |
| Zoom リンク・参加 URL | 中（URL が流出しても損害は限定的） |

## 5. Lark API のセキュリティ

**⚠️ 設計次第で安全・不安全が大きく変わる**

Lark API は以下のトークンを使います：

| トークン種別 | 有効期限 | 権限範囲 |
|------------|---------|---------|
| `tenant_access_token` | 約2時間（自動更新） | **テナント全体**（全データにアクセス可能） |
| `user_access_token` | 約2時間（リフレッシュ可） | 特定ユーザーの権限のみ |

### 必須の設計ルール

```
❌ NG: フロントエンド（ブラウザ）に tenant_access_token を渡す
✅ OK: サーバーサイドのみで tenant_access_token を使用し、結果だけをブラウザに返す
```

現在の Living Me の設計（Next.js Server Component / Server Actions からのみ Lark API を呼ぶ）は正しい実装です。

### トークン管理ルール

- `LARK_APP_SECRET` は **Vercel 環境変数に保存し、コードには書かない**
- ソースコード（GitHub）に App Secret を絶対に commit しない
- `.gitignore` に `.env.local` が含まれていることを確認する

## 8. 推奨事項

### 現在の設計で継続してよいこと

- ✅ **個人情報は PostgreSQL（Neon）に保存**するアーキテクチャは正しい
- ✅ **Lark Base には公開コンテンツ（イベント情報・アーカイブ・コラム）のみを保存** する
- ✅ **Next.js サーバーサイドからのみ Lark API を呼ぶ**（`tenant_access_token` をブラウザに渡さない）

### 追加で対応すべきこと

| 優先度 | 対応 | 理由 |
|--------|------|------|
| 🔴 必須 | `LARK_APP_SECRET` を Vercel 環境変数に設定し、GitHub に push しない | API 認証情報の漏洩防止 |
| 🔴 必須 | 会員の氏名・メール・ジャーナルなど個人情報を Lark Base に入力しない | ByteDance リスク・APPI 対応 |
| 🟡 推奨 | サービスのプライバシーポリシーに「コンテンツデータを Lark（ByteDance 子会社）のサーバーで処理する」旨を明記 | APPI 第三者提供の透明性確保 |
| 🟡 推奨 | Lark Base データを定期的にエクスポートしてバックアップ | ベンダーロックイン・障害対策 |
| 🟢 任意 | ByteDance リスクが懸念される場合は Notion API + データベース連携への移行を検討 | 親会社リスクの排除 |

### Lark Base に保存してよいデータ・してはいけないデータ

```
✅ 保存してよいデータ
  - イベントのタイトル・説明・日時・場所
  - アーカイブのタイトル・カテゴリ・動画URL
  - 主宰者コラムの文章
  - 今日のエネルギーシェア・ジャーナリングテーマ
  - タグ・カテゴリのマスタデータ

❌ 保存してはいけないデータ
  - 会員の氏名・メールアドレス・Discord ID
  - 会員のジャーナル（個人の日記）
  - 決済情報・Stripe の顧客データ
  - セッション・認証トークン
  - パスワード・秘密鍵などの認証情報
```
