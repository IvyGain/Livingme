## 3. セキュリティ要件

### 3.1 認証・認可

| 要件 | 内容 |
|------|------|
| 認証方式 | （Discord 連携）|
| セッション管理 | JWT / サーバーサイドセッション |
| 権限制御 | ミドルウェアレベルでのルート保護 |
| CSRF対策 | Next.js標準CSRF保護 |

### 3.2 決済セキュリティ

- Stripe Webhook の署名検証（`stripe-signature` ヘッダー）
- カード情報は一切サーバーに保存しない
- PCI DSS 準拠（Stripe委譲）

### 3.3 データ保護

- 個人情報: パスワードのハッシュ化（bcrypt）
- 通信: HTTPS強制（HSTS）
- 環境変数: シークレット情報は環境変数管理（`.env`はgit管理外）
- OWASP Top 10 への対応

### 3.4 アクセス制御

- 会員ステータス別コンテンツ保護（trial / member / inactive）
- 管理画面はスタッフ権限のみアクセス可
- Stripe Webhook は署名検証済みリクエストのみ処理

---

## 4. 運用性要件

### 4.1 管理画面

- **非エンジニアでも更新できる管理画面**（最優先）
- 朝会がない日の手動更新が簡単に行える
- 動画登録 / タグ付け / 再生リスト管理が容易
- イベント企画時に申し込みフォームをサイト内で作成可能
- フォーム回答を管理画面から閲覧可能
- Discord連携によるイベント用専用チャット自動作成

### 4.2 コンテンツ更新フロー

- 朝会動画登録後、AI抽出 → プレビュー → 公開の3ステップ以内
- 手動更新時の操作は5分以内に完結できる設計

### 4.3 監視・ログ

- ログイン履歴などをDBに収集

## 11. 技術的検討事項

| 検討事項 | 方針 |
|---------|------|
| Stripe-Discord-サイト権限の自動同期 | Webhook受信後の冪等性設計・リトライキュー |
| 混在アーカイブ管理 | Lark / Google Drive / 手動アップを統一インターフェース |
| 朝会議事録からのトップページ自動更新 | AI抽出パイプライン（Claude API等）設計 |
| 3,000〜5,000名規模 | DB設計・CDN・Redisキャッシュ戦略 |
| 非エンジニア運用CMS | 管理画面UX最優先設計 |
| trial / inactive / member 権限制御 | Next.js middleware レベルでの実装 |
| 紹介制度の報酬計算 | 月次バッチ処理（Cron）推奨 |
| 提携アンバサダー6ヶ月判定 | 入会日からの継続月数の自動計算ロジック |
| AI機能追加前提の構成 | モジュラーAPI設計・拡張ポイントの明示 |
